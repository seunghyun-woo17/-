## 버그 (수정 필요)

### ✅ BUG-01: PDF 출력 시 항상 마지막 PO 데이터만 사용 [수정완료 2026-04-18]
- **위치**: `collectPOData()` 함수 (line 1373)
- **코드**: `const po = DB.po_header[DB.po_header.length - 1];`
- **증상**: PO-A, PO-B 두 개 발행 후 PO-A의 "PDF 발행" 버튼을 눌러도 PO-B 내용이 출력됨
- **영향도**: 높음 — 잘못된 PO 문서 발행 가능

### ✅ BUG-02: 단가(Unit Price) 입력 필드 없음 [수정완료 2026-04-18]
- **위치**: PO 생성 폼 + `buildPODocHTML()` (line 1384)
- **코드**: `l.unit_price || 5000000` (단가 미입력 시 500만원 고정)
- **증상**: PO 문서의 모든 품목 단가가 "5,000,000원"으로 고정 출력됨
- **영향도**: 높음 — 실제 계약 금액과 다른 문서 발행

### 🟡 BUG-03: 검수자(Inspector) 이름 하드코딩
- **위치**: `completeIncoming()` (line 1186)
- **코드**: `inspector: 'CHOI SANG DEOI'`
- **증상**: 모든 입고 리포트 검수자가 'CHOI SANG DEOI'로 고정
- **영향도**: 중간 — 다른 직원이 검수해도 같은 이름이 기록됨

### 🟡 BUG-04: PO 조건 번호 순서 오류 (2번 누락)
- **위치**: `buildPODocHTML()` 하단 조건 텍스트 (line 1454)
- **코드**: 1번 → 3번 → 4번 → 5번 (2번 없음)
- **증상**: PO 문서 하단 조건 목록에서 2번 항목이 빠져 있음
- **영향도**: 낮음 — 문서 품질 문제

### 🟡 BUG-05: Incoming Report MODEL 컬럼 하드코딩
- **위치**: `refreshIncoming()` (line 1326)
- **코드**: `<td>HiNAS Navigation</td>` (모든 품목에 동일)
- **증상**: 다른 제품을 입고해도 리포트에는 "HiNAS Navigation"으로 표시
- **영향도**: 중간 — 품목이 다양해질 경우 오류

### 🟡 BUG-06: 미리보기 모달과 저장 파일 내용 불일치 가능
- **위치**: `printPODoc()` vs `buildPODocHTML()` 두 함수
- **증상**: 모달 미리보기(폼 값 직접 읽기)와 실제 저장 파일(DB에서 읽기)이 다른 함수를 사용해 내용이 다를 수 있음. 특히 납기일(Due Date)이 모달에는 없고 저장 파일에만 있음.
- **영향도**: 중간

## 기능 개선 필요 사항

### ⚠️ IMPROVE-01: 외부 CDN 의존 (오프라인 불가)
- jsQR 라이브러리를 외부 CDN에서 로드 (line 354)
- 인터넷 연결 없는 사내망 환경에서 QR 스캔 기능 불가
- **조치**: 라이브러리 파일을 내부에 포함하거나 npm 패키지로 관리

### ⚠️ IMPROVE-02: PDF 저장이 실제 PDF가 아닌 HTML 파일
- "PDF 발행" 버튼이 실제 PDF가 아닌 .html 파일로 저장
- 사용자가 수동으로 브라우저 인쇄 → "PDF로 저장" 선택해야 함
- File System Access API는 Chrome/Edge에서만 지원 (Safari, Firefox 불가)
- **조치**: 서버 이전 시 실제 PDF 생성 라이브러리 (예: Puppeteer, wkhtmltopdf) 사용

### ⚠️ IMPROVE-03: 회사 주소 두 군데 불일치
- 헤더/상단: `11F, 70, Nonheon-ro 85-gil, Gangnam-gu, Seoul` (서울 강남)
- PO 문서 발신자: `7F, 7-12, Jungang-daero 865beon-gil, Dong-gu, Busan` (부산 동구)
- **조치**: 정확한 공식 주소 확인 후 통일

### ⚠️ IMPROVE-04: "업체 자동 통보 발송" — 실제 이메일 발송 없음
- 수량 부족 입고 시 "업체 자동 통보 발송"이라고 UI에 표시
- 실제 이메일 발송 로직 전혀 없음 (표시만 됨)
- **조치**: 서버 이전 시 이메일 발송 API 연동 (SMTP or SendGrid 등)

### ⚠️ IMPROVE-05: PO 수정/취소 기능 없음
- 발행된 PO에 오류가 있어도 수정 또는 취소 방법이 없음
- 현재는 DB 전체 초기화(리셋)만 가능
- **조치**: PO 편집, PO 상태 변경(CANCELLED 등) 기능 추가

### ⚠️ IMPROVE-06: 동시에 하나의 PO만 입고 처리 가능
- 스캔 탭에서 세션이 하나만 유지됨
- 여러 업체가 동시에 납품하는 상황 처리 불가
- **조치**: 서버 이전 시 사용자별 세션 관리로 해결 가능

### ⚠️ IMPROVE-07: 새로고침 시 전체 데이터 소멸
- 작업 중 실수로 F5를 누르면 입고 진행 중인 스캔 데이터 전부 소멸
- **조치**: 서버 이전 시 DB 연동으로 자동 해결

### ⚠️ IMPROVE-08: PO 문서 Terms & Conditions 하드코딩
- PO 조건 텍스트 (납품 조건, 결제 조건 등) 모두 고정값
- `"MRC 보관(재고 관리) 및 호선별 납품 조건"`, `"선금 30% / 잔금 70%"` 하드코딩
- **조치**: PO별로 입력 가능하도록 필드 추가
